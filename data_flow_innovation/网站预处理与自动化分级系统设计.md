# 金融公司网站预处理与自动化分级系统设计

> **核心目标**：在正式数据采集之前，对每一个金融公司网站进行全方位"体检"，
> 输出结构化的网站画像（Site Profile），并自动打上 **AI 可全自动 / 辅助自动 / 需人工** 三档标签，
> 为后续采集任务的路由调度提供决策依据。

## 二、阶段 0：URL 规范化与去重

```
输入：任务大厅下发的金融公司 URL（可能格式不统一）

处理步骤：
  ① 统一 URL 格式（scheme 补全、去除 trailing slash、小写化域名）
  ② 域名去重（同一公司多个 URL 合并为主域名）
  ③ 解析公司基本信息：
       - 域名后缀 → 推断注册国家/地区（.hk, .co.jp, .fr, .de, .it...）
       - 公司名称 → 匹配已知金融机构数据库（可用 Wikidata / OpenCorporates）
  ④ 检查黑名单（已知无法访问的域名直接标红，不浪费分析资源）

输出：标准化 URL 列表 + 初始元数据
```

## 三、阶段 1：轻量探针（30 秒快检）

每个 URL 并发发起快速探测，无需完整渲染，目标是在最短时间内完成初步分类过滤。

### 1.1 HTTP 元数据探针

```python
# 使用 httpx 异步发送 HEAD/GET 请求（不渲染 JS）
探测项：
  - HTTP 状态码（200 / 301 / 403 / 503...）
  - 响应头分析：
      Server: cloudflare → 标记 CF_DETECTED
      cf-ray:            → 确认 Cloudflare
      x-frame-options    → 安全策略强度
      set-cookie         → Session 机制探测
      content-language   → 语言初判
  - TLS 证书信息（颁发机构、有效期）
  - 响应时间（> 5s → 可能有反爬限速）
  - robots.txt 获取与解析（Disallow 规则影响策略选择）
```

### 1.2 Cloudflare 威胁等级快速判断

```
响应特征                           → Cloudflare 等级          → 处理策略
────────────────────────────────────────────────────────────────────────
HTTP 200, 无 cf-ray               → 无 CF 防护               → GREEN 候选
HTTP 200, 有 cf-ray, 内容正常      → CF 代理但未启用质询       → GREEN 候选  
HTTP 403/503, "Checking browser"  → CF Under Attack Mode      → YELLOW 候选
HTTP 403, JS Challenge 页面        → CF JS Challenge           → YELLOW 候选
HTTP 403, Turnstile CAPTCHA        → CF Turnstile              → YELLOW/RED
需登录才能访问 IR 页面              → 企业级访问控制            → RED 候选
```

### 1.3 快检输出（JSON 结构）

```json
{
  "url": "https://example-financials.co.jp",
  "quick_check": {
    "reachable": true,
    "status_code": 200,
    "response_time_ms": 842,
    "cloudflare_detected": true,
    "cloudflare_level": "proxy_only",
    "robots_restricted": false,
    "inferred_language": "ja",
    "inferred_country": "JP",
    "quick_grade": "GREEN_CANDIDATE"
  }
}
```

## 四、阶段 2：深度分析 Agent 团队

通过轻量探针后，对每个 URL 启动深度分析。使用 **Playwright headless browser** 完整渲染页面，
并由多个专项 Agent 并行分析不同维度。

### 2.1 技术栈探针 Agent

**目标**：识别网站前端框架与渲染模式，决定爬取策略

```
检测项                    检测方法                        影响
────────────────────────────────────────────────────────────────────────────
服务端渲染(SSR/静态HTML)   首屏 HTML 含实质内容             最易自动化 ✅
SPA (React/Vue/Angular)   首屏 HTML 为空壳, JS 注水        需等待 JS 执行 ⚠️
PDF 内嵌 Viewer           <iframe> + PDF.js / Adobe       需特殊提取策略 ⚠️
音频嵌套播放器             <video>/<audio>/第三方播放器     需解析媒体 URL ⚠️
JavaScript 跳转链接        onclick / window.open           需执行 JS ⚠️
后台分页 / 无限滚动        Intersection Observer / XHR     需模拟滚动 ⚠️
第三方文档托管平台          Issuu / DocuSign / Box          需平台专属适配 ❌

技术栈识别方式：
  - window.__NEXT_DATA__   → Next.js
  - window.__nuxt__        → Nuxt.js  
  - ng-version attribute   → Angular
  - __vue__                → Vue
  - 响应头 X-Powered-By    → 后端框架
```

### 2.2 IR 页面深度导航 Agent

**目标**：找到并评估"投资者关系"入口的可访问性与结构复杂度

```
导航探索策略（Tree-of-Thought 多路径并行）：

  路径 A：主导航栏关键词匹配（视觉 + 文本双通道）
  路径 B：页面底部 Footer 链接扫描（IR 链接常在此处）
  路径 C：Sitemap.xml 解析（/sitemap.xml, /sitemap_index.xml）
  路径 D：Google Site Search API 预搜索
           site:{domain} "annual report" | "年度报告" | "年次報告"

评估指标：
  IR页面导航深度（层数）   1层(主页直达) → GREEN
                           2-3层          → YELLOW  
                           > 3层 / 无法找到 → RED

  IR页面内容丰富度：
    - 年报列表是否分年份归档
    - 季报是否单独分类
    - 业绩发布会音频/视频是否存在
    - 历史数据可追溯年限

  GPT-4o Vision 综合评分输出（0-100分）：
    导航清晰度、内容完整性、链接可提取性
```

### 2.3 反爬机制深度探测 Agent

**目标**：全面评估网站的反爬强度，输出精确的对抗策略

```
探测维度                  探测方法                           结果标记
────────────────────────────────────────────────────────────────────────
Cloudflare 等级           模拟爬虫 UA 请求，观察响应          level_0-5
Rate Limiting            短时高频请求，观察 429/503           ratelimit_ms
Honeypot 链接             DOM 中隐藏链接检测（display:none）   honeypot_detected
User-Agent 检测           普通UA vs 浏览器UA 对比响应差异     ua_sensitive
IP 信誉检测               数据中心 IP vs 住宅 IP 对比         ip_sensitive
TLS 指纹检测 (JA3)        使用 curl-impersonate 测试          tls_sensitive
行为指纹检测              鼠标移动/点击间隔分析               behavior_sensitive
登录墙 (Login Wall)       IR 页面是否强制登录                 login_required
地区封锁 (GeoBlock)       模拟不同 IP 地区访问                geo_restricted

综合反爬强度评分：
  0-2 分 → 基本无防护      → AI 全自动可行
  3-5 分 → 中等防护        → 需 stealth + 代理配合
  6-8 分 → 强防护          → 需 AI Vision + 住宅 IP + 行为模拟
  9-10分 → 极强防护/登录墙  → 人工介入必要
```

### 2.4 链接模式分析 Agent

**目标**：识别并记录该网站 PDF/音频链接的分布规律，建立采集模板

```
分析项：

  文件链接特征：
    - 直接 PDF URL（<a href="...pdf">）            → 最易采集 ✅
    - API 接口返回（XHR/Fetch 动态加载）           → 需抓 Network 请求 ⚠️
    - 第三方 CDN（akamai / cloudfront / fastly）   → 有签名 URL 限制 ⚠️
    - 第三方平台（Bloomberg / Refinitiv / S&P）    → 需平台特殊处理 ❌
    - 内嵌文档查看器（无法直接下载）               → 需截图/打印 ❌

  音频/视频链接特征：
    - 直接 MP3/MP4 URL                             → 最易采集 ✅
    - YouTube/Vimeo 嵌入                           → yt-dlp 处理 ✅
    - 企业直播平台（Webcast / Lumi / Chorus.ai）   → 需专属适配 ⚠️
    - Flash/Legacy Player（极少数老旧网站）          → 需人工介入 ❌

  URL 规律提取（用于后续自动采集模板）：
    pattern_example: "https://ir.company.com/reports/{year}/annual-report.pdf"
    regex_template:  "https://ir\\.company\\.com/reports/\\d{4}/annual-report\\.pdf"

  历史链接数量统计（验证数据完整性基准）
```

### 2.5 多语言能力检测 Agent

```
检测项：
  - 页面主语言（html lang 属性 + 内容识别）
  - 是否有语言切换功能
  - 切换后 URL 规律（/en/ /ja/ /zh-hk/ 等路径变化）
  - IR 内容是否有多语言版本（年报有无英文版）

输出：
  language_profile: {
    primary: "ja",
    available: ["ja", "en"],
    switch_pattern: "?lang={code}",
    ir_english_available: true
  }

策略含义：
  仅有非英语且无英文切换 → 必须使用多语言 LLM 理解 → 标记 MULTILANG_REQUIRED
  有英文版               → 优先切换至英文版采集      → 降低复杂度
```

## 五、阶段 3：AI 分级决策引擎

综合以上所有维度的探测结果，由 LLM 进行最终判定，输出三档分级标签。

### 5.1 分级标准矩阵

```
                      反爬强度 0-2   反爬强度 3-5   反爬强度 6-8   反爬强度 9-10
────────────────────────────────────────────────────────────────────────────────
IR 导航 1层  + 直链    ██ GREEN       ██ GREEN       █ YELLOW       ░ RED
IR 导航 2层  + 直链    ██ GREEN       █ YELLOW       █ YELLOW       ░ RED
IR 导航 3层  + 动态    █ YELLOW       █ YELLOW       ░ RED          ░ RED
IR 导航 4层+ / 无IR    ░ RED          ░ RED          ░ RED          ░ RED
第三方平台托管         █ YELLOW       █ YELLOW       ░ RED          ░ RED
登录墙 / 地区封锁      ░ RED          ░ RED          ░ RED          ░ RED
```

### 5.2 三档标签定义与处置策略

```
┌─────────────────────────────────────────────────────────────────────┐
│  🟢 GREEN — AI 全自动采集                                            │
│                                                                      │
│  特征：                                                              │

│    · IR 页面导航层数 ≤ 2 层                                          │

┌─────────────────────────────────────────────────────────────────────┐
│  🟡 YELLOW — AI 辅助自动采集（需增强策略）                           │
│                                                                      │
│  特征：                                                              │

│  🔴 RED — 需人工介入                                                  │

│    · 反爬强度 9-10 或企业级登录墙                                    │
│    · IR 内容完全托管于第三方无法适配平台                              │
│    · 地区封锁（无代理可绕过）                                         │
│    · 网站结构无法被 AI 理解（极度非标准）                             │
│    · 文件仅通过 Email 申请获取                                        │
│                                                                      │
│  处置：                                                              │
│    · 生成人工介入工单（含网站截图、已尝试策略列表）                   │
│    · 在 Dashboard 高亮标记，推送 Slack/钉钉通知                      │
│    · 人工处理后将操作录制为 Recipe，供后续自动化学习                  │
│  预计占比：约 10-15% 的金融公司网站                                  │
```
